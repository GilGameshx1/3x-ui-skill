## 🔧 Как внести вклад

### 1. Форк репозитория

Нажмите **Fork** на GitHub, чтобы создать копию репозитория в вашем аккаунте.

### 2. Клонируйте репозиторий

```bash
git clone https://github.com/YOUR_USERNAME/3x-ui-skill.git
cd 3x-ui-skill
```

### 3. Создайте ветку

```bash
git checkout -b feature/your-feature-name
```

Используйте префиксы для названий веток:
- `feature/` — для новых функций
- `bugfix/` — для исправления ошибок
- `docs/` — для документации
- `refactor/` — для рефакторинга

### 4. Внесите изменения

Сделайте нужные изменения в коде или документации.

### 5. Протестируйте

Запустите тестовый скрипт:
```bash
./test.sh
```

### 6. Закоммитьте изменения

```bash
git add .
git commit -m "feat: описание изменений"
```

### 7. Отправьте в репозиторий

```bash
git push origin feature/your-feature-name
```

### 8. Создайте Pull Request

Откройте PR на GitHub и опишите ваши изменения.

## 📝 Стандарты кода

### Shell-скрипты

- Используйте `#!/bin/bash` или `#!/usr/bin/env bash`
- Всегда используйте `set -euo pipefail` для безопасности
- Используйте функции для повторяющейся логики
- Комментируйте сложные участки кода
- Используйте понятные имена переменных

**Пример:**
```bash
#!/usr/bin/env bash
set -euo pipefail

# Цвета для вывода
RED='\033[0;31m'
GREEN='\033[0;32m'
NC='\033[0m'

# Функция проверки
check_something() {
    if [ condition ]; then
        echo -e "${GREEN}✓ Успех${NC}"
        return 0
    else
        echo -e "${RED}✗ Ошибка${NC}"
        return 1
    fi
}
```

### Документация (Markdown)

- Используйте заголовки `##`, `###`, `####`
- Добавляйте emoji для визуальной навигации
- Используйте таблицы для сравнений
- Добавляйте примеры кода с подсветкой

### Сообщения коммитов

Используйте [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` — новая функция
- `fix:` — исправление ошибки
- `docs:` — документация
- `style:` — форматирование
- `refactor:` — рефакторинг
- `test:` — тесты
- `chore:` — служебные изменения

**Примеры:**
```bash
feat: добавить скрипт восстановления БД
fix: исправить путь копирования в install.sh
docs: перевести QWEN.md на русский
refactor: оптимизировать проверку зависимостей
```

## 🔄 Процесс Pull Request

1. **Создайте PR** с понятным названием
2. **Опишите изменения** в описании PR
3. **Добавьте скриншоты** (если применимо)
4. **Пройдите тесты** (если настроены CI)
5. **Дождитесь ревью** от мейнтейнеров
6. **Внесите правки** (если требуется)
7. **Получите аппрув** и слияние

### Шаблон Pull Request

```markdown
## Описание
Краткое описание изменений

## Тип изменений
- [ ] Новая функция
- [ ] Исправление ошибки
- [ ] Документация
- [ ] Рефакторинг

## Тестирование
- [ ] Запущен test.sh
- [ ] Протестировано на реальном сервере

## Чеклист
- [ ] Код соответствует стандартам
- [ ] Документация обновлена
- [ ] CHANGELOG обновлён (если нужно)
```

## 🐛 Сообщение об ошибках

### Перед созданием issue

1. **Проверьте существующие** — возможно, ошибка уже известна
2. **Проверьте последнюю версию** — возможно, уже исправлено
3. **Соберите информацию**:
   - Версия навыка
   - Версия Qwen Code
   - ОС сервера
   - Логи ошибок

### Шаблон отчёта об ошибке

```markdown
## Описание ошибки
Краткое описание проблемы

## Воспроизведение
Шаги для воспроизведения:
1. Запустить команду '...'
2. Ввести данные '...'
3. Увидеть ошибку

## Ожидаемое поведение
Что должно было произойти

## Логи
```
Текст ошибки или скриншот
```

## Окружение
- Версия навыка: 2.0.0
- ОС: Ubuntu 22.04
- Qwen Code: x.x.x

## Дополнительная информация
Любые другие детали
```

## 💡 Запрос функций

### Перед созданием запроса

1. **Проверьте существующие** — возможно, функция уже запрошена
2. **Обдумайте необходимость** — действительно ли это нужно всем
3. **Опишите_use case** — как и зачем использовать

### Шаблон запроса функции

```markdown
## Описание функции
Что вы хотите добавить

## Проблема
Какую проблему решает эта функция

## Предлагаемое решение
Как это должно работать

## Альтернативы
Какие есть обходные пути

## Дополнительная информация
Примеры использования, скриншоты, ссылки
```
